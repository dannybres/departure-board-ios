FEATURE: Lock Screen Widget Plan (building on existing Home Screen widgets)
=======================================================================

SUMMARY
-------
Home Screen widgets already exist in this project. This plan adds Lock Screen
widget families with commuter-focused, low-noise glanceable info:
- accessoryInline
- accessoryCircular
- accessoryRectangular

Goal: deliver fast-to-read "next train" status on Lock Screen while keeping
home widget behavior unchanged.

PRODUCT GOALS
-------------
1) Glanceability in < 1 second
   - User should understand next departure state without unlocking.
2) Reliability over density
   - Prefer one clear item over too much compressed text.
3) Minimal setup friction
   - Reuse current station/board AppIntent selection model.
4) Consistent deep links
   - Tap opens the same station board route used by current widgets.

WIDGET DEFINITIONS
------------------

1. accessoryInline (single-line context)
   Primary output examples:
   - "14:32 → Waterloo On time"
   - "14:32 → Waterloo +7m"
   - "14:32 → Waterloo Cancelled"

   Content rules:
   - One service only (next non-departed service).
   - Truncate destination aggressively.
   - If no services: "No departures soon".

2. accessoryCircular (time-first)
   Primary output:
   - Center: scheduled or expected time (e.g. "14:32")
   - Footer/small text: status shorthand ("On time", "+7", "Canx")

   Content rules:
   - One service only.
   - Use symbolic color semantics where available, but ensure text still
     communicates state in monochrome environments.

3. accessoryRectangular (compact list)
   Primary output:
   - Header: station short name
   - Rows: top 2 departures max

   Row shape:
   - time | destination (truncated) | status/platform token

   Content rules:
   - Prefer showing 2 useful rows over 3 cramped rows.

CONFIGURATION MODEL
-------------------
Reuse `SingleStationIntent` for lock-screen widget configuration so users can
pick the same board source (favourite, departures/arrivals, filtered board,
nearest station sentinel).

Do NOT enable lock-screen families for dual-station widget; readability is too
poor for split content.

TECHNICAL PLAN
--------------

A) Widget registration
- File: `DepartureBoardWidget/DepartureBoardWidgetBundle.swift`
- Add a dedicated `LockScreenBoardWidget` in the bundle.
- Keep existing home widgets intact.

B) Widget declaration
- File: `DepartureBoardWidget/DepartureBoardWidget.swift`
- Add `LockScreenBoardWidget` using `AppIntentConfiguration` +
  `SingleStationProvider`.
- `supportedFamilies`:
  - `.accessoryInline`
  - `.accessoryCircular`
  - `.accessoryRectangular`

C) Family-specific views
- Add 3 focused view components:
  - `InlineBoardView`
  - `CircularBoardView`
  - `RectangularBoardView`
- Add shared formatter helper for status shorthand and destination truncation.

D) Data slicing strategy
- Keep provider fetch size unchanged or moderately reduced.
- Slice in view layer per family:
  - Inline: first 1 service
  - Circular: first 1 service
  - Rectangular: first 2 services

E) Deep-link behavior
- Entire accessory widget opens the configured station board.
- Keep route parity with current home-screen widget deep links.

UX & CONTENT RULES
------------------
- Status priority:
  1) Cancelled
  2) Delayed (+Xm)
  3) On time
- Prefer expected/estimated time when materially changed.
- Avoid verbose wording in Lock Screen contexts.
- Always provide a meaningful empty/error message:
  - Empty: "No departures soon"
  - Error: "Board unavailable"

ACCESSIBILITY
-------------
- Ensure Dynamic Type clipping is controlled with sensible line limits.
- Avoid relying only on color to convey service state.
- Keep monospaced time text for fast scanning.

PHASED DELIVERY
---------------
Phase 1 (MVP, ~1 day)
- Add `accessoryInline` + `accessoryRectangular`.
- Reuse existing data models and intent configuration.
- Ship with simple status labels and stable deep links.

Phase 2 (~0.5-1 day)
- Add `accessoryCircular` time-first design.
- Tune truncation rules and status shorthand.

Phase 3 (polish, optional)
- Family-specific placeholders and snapshots.
- Fine-grained typography tuning for long destination names.

ACCEPTANCE CRITERIA
-------------------
1) User can add a lock-screen widget and configure station/board via intent.
2) Inline widget shows one next service with readable state.
3) Rectangular widget shows up to two services without overlap/clipping.
4) Circular widget shows one service with state shorthand.
5) Tapping each widget opens the correct in-app board.
6) Empty/error states are user-readable (not blank).

TEST CHECKLIST
--------------
- Device lock-screen preview for all 3 accessory families.
- Long destination stress test (truncation behavior).
- Delay/cancelled/on-time states for each family.
- Nearest-station sentinel configuration path.
- Tap-through deep link correctness.

OUT OF SCOPE (for this plan)
----------------------------

- Push disruption notifications.
- Dual-board lock-screen layout.
